Token economics

Most tokens go toward orientation, not work.

Every time an agent starts a task, it reads the project for context. On a large codebase, that orientation is expensive—and it repeats every session, every developer, every query.

The agent isn't slow because the model is slow. It's slow because the model is lost.

Grep → read Grep → read Grep → read Repeat
The majority of an agent's token budget goes toward figuring out where things are and how the code is structured—not solving your actual problem.
Real-world example

A 200-developer team rolled back their AI coding tool deployment after token overages hit $22K/month. 70% of the cost came from 30 developers working on legacy codebases.

Current approaches

How agents build context today.

1
File scanning

Agents read files, follow references, explore structure. High-token, repeated across sessions, only provides partial visibility.

2
Documentation

READMEs, wikis, comments. Often outdated, incomplete, disconnected from how the code actually behaves.

3
Context retrieval

RAG, MCP, tools. Pull snippets per query. Inferred rather than resolved, rebuilt every interaction, constrained by token limits.

All three force the agent to approximate understanding. More tokens. More time. More probabilistic outcomes.
The irony

The knowledge agents need is already in your codebase.

Architectural boundaries. Dependency relationships. Security conventions. Organizational standards. All encoded in structure, configuration, dependencies, and patterns.

The problem isn't that this knowledge is missing. It's that agents don't have a way to see and use it.

What if you could extract it once? Structure it for agents? Version it with the code?
Part 2

What Prethink discovers.

Moderne Prethink

Deterministic code knowledge from compiler-accurate analysis.

Prethink generates structured context from Moderne's Lossless Semantic Trees—the same code model that has powered OpenRewrite's automated transformations for years.

Every symbol, type, method call, configuration reference, and dependency is resolved to what it actually refers to. Not an AST. Not an embedding. A fully resolved model of the repository.

Not MCP Not RAG Not embeddings Not prompt engineering
Those approaches retrieve or infer context at runtime. Prethink resolves it in advance, deterministically, from the code itself.
Output location

Context lives in the repository next to the code. Versioned, reviewable, committed alongside. It's just a recipe—so it could also push to a central context service if that becomes the standard.

What it finds

What an experienced developer knows, made explicit.

Service interfaces

Endpoints, database connections, external calls, messaging patterns. Impact and integration boundaries resolved.

Dependency intelligence

Complete dependency trees including transitive dependencies. Version relationships for upgrade risk and compatibility.

Test coverage & gaps

Security & configuration

Auth setup, access controls, CORS, OAuth2. Agents see security configuration without inferring from scattered files.

Conventions & error handling

Naming patterns, import organization, exception strategies. Agents write code that fits the existing style.

Architecture & deployment

Part 4

Code quality metrics.

Quality intelligence agents can read—not dashboards they never visit.

The shift

Quality intelligence belongs where the work happens.

Experienced developers carry quality judgment in their heads—which code is fragile, where the untested risks are, which classes have too many responsibilities. Agents need that same awareness, in a form they can consume.

Prethink puts quality metrics wherever your agents will read them. Default is committed to the repo—reviewable, versioned, no extra system to call. But because it's just recipe output, the same data can flow to a central context store, an MCP-served knowledge base, or wherever fits your architecture.

Quality metrics

Metrics at every level, materialized for agents.

Method level

Cyclomatic & cognitive complexity. Nesting depth. ABC decomposition (orchestrator, transformer, or decision-maker?). Halstead estimated bugs. Composite debt score.

Class level

LCOM4 cohesion—“this class should be split into 3 classes.” Tight class cohesion (TCC). Coupling (CBO). Weighted methods (WMC). Maintainability index.

Package level

Afferent/efferent coupling. Instability & abstractness. Dependency cycle detection. In one analysis, nearly half of all packages had circular dependencies.

Code smell detection

God Class, Feature Envy, Data Class—with severity ratings and metric evidence, not just flags.

Test gap analysis

Untested methods ranked by risk score (complexity × call count). Trivial methods filtered out. Suggested test class for each gap.
